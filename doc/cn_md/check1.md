## 0. 概述

在检查点 0 中，你使用互联网流套接字从网站获取信息并发送电子邮件，这一切基于 Linux 内置的 TCP 实现。TCP 将不可靠的数据报转化为一对可靠的、有序的字节流。你也在内存中实现了字节流抽象。在接下来的几周里，你将亲自实现 TCP，在两台由不可靠数据报网络连接的计算机之间提供字节流抽象。

> **为什么我要这样做？** 在更不可靠的服务之上提供一个抽象或服务，是网络领域许多有趣问题的核心。过去 40 年，研究者和工程师们已经找到了如何通过互联网传输各种内容——消息和电子邮件、超链接文档、搜索引擎、音视频、虚拟世界、协作文件共享、数字货币等。TCP 本身的作用——使用不可靠的数据报提供一对可靠的字节流——就是最经典的例子之一。

## 2. 实践部分：班级专用网络

我们为 CS144 课程创建了一个专用网络，让你的 VM 可以直接与班上其他同学的 VM 收发数据报。

**加入网络**：
1. 安装 wireguard：`sudo apt install wireguard`
2. 访问 https://cs144.net/wg 并按照说明加入 CS144 专用网络
3. 加入后，按照页面上的"ping"说明验证连接
4. 每次重启 VM 后都需要重新加入网络

### 2.1 Ping 同学并查看数据报

1. 在你自己的电脑（不是 VM）上安装 Wireshark
2. 向同学请求他们的 IP 地址，使用 ping 命令发送"回显请求"数据报
3. 提示：
   - 按 `Ctrl-C` 结束 ping
   - 使用 `-i 0.2` 让 ping 每 0.2 秒发送一次请求
   - 另一个终端运行 `killall -QUIT ping` 查看统计信息

4. 在 writeup 中记录：
   - VM 到同学的往返延迟平均值
   - 交付率（收到回显回复的百分比）和丢失率
   - 是否看到重复的数据报（ping 会打印"DUP"）
   - 使用 `sudo tcpdump -n -w /tmp/capture.raw -i wg0` 捕获数据报
   - 用 Wireshark 检查 `/tmp/capture.raw` 文件，找到 IP 数据报的各个字段

### 2.2 手动发送互联网数据报

在 `apps/ip_raw.cc` 中编写程序，使用原始套接字向同学发送互联网数据报。

1. 发送 IP 协议为"5"的数据报给同学（需要用 sudo 运行）
2. 发送 UDP 数据报（IP 协议"17"），同学可以用 `nc -u` 接收
3. 将代码包含在提交中

## 3. 实现：将子字符串按顺序组装

作为实验的一部分，你需要实现一个 **TCPReceiver**：接收数据报并将其转换为可靠的字节流，供应用程序从套接字读取。

TCP 发送方将字节流分割成短段（每个不超过约 1460 字节），但网络可能会对这些数据报进行重排、丢失或重复交付。接收方必须将这些段重新组装成原始的连续字节流。

你的任务是实现一个 **Reassembler** 数据结构。

### 接口

```cpp
void insert(uint64_t first_index, std::string data, bool is_last_substring);
uint64_t count_bytes_pending() const;
Reader& reader();
```

### 3.1 Reassembler 内部应该存储什么？

Reassembler 需要处理三类知识：

1. **流的下一个字节**：一旦知道，就立即推送到 ByteStream
2. **落在流容量范围内但尚不能写入的字节**（因为前面的字节仍然未知）：存储在 Reassembler 内部
3. **超出流容量的字节**：丢弃

容量限制的是：
- ByteStream 中缓冲的字节数（已组装）
- "未组装"子字符串使用的字节数

### 3.2 FAQ

- 流中第一个字节的索引是什么？**零**
- 效率要求？选择合适的数据结构。基准测试要求超过 0.1 Gbit/s
- 重复的子字符串如何处理？不存储重叠的冗余信息
- 需要多少行代码？Reassembler 约 50-60 行
